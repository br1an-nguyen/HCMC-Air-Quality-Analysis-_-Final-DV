## Phần 1 — Vấn đề kỹ thuật DOM Injection

🔴 **Critical: `window.parent` bị block bởi same-origin policy**

Kế hoạch đề xuất inject qua `window.parent.document.body` trong `st.components.v1.html()`. Streamlit render component trong iframe sandbox ở một port phụ (3001). Truy cập `window.parent` từ iframe sang app chính bị block vì khác origin.

**Plaintext**

```
iframe origin: http://localhost:3001
app origin:   http://localhost:8501
→ Cross-origin → window.parent.document BLOCKED
```

> **GIẢI PHÁP ĐÚNG:**
>
> Dùng `st.components.v1.html(html_str, height=700)` với height đủ lớn. Widget float bên trong iframe viewport — không tràn ra ngoài, nhưng với height lớn thì người dùng không nhận ra sự khác biệt. Đây là cách mọi Streamlit custom component thực sự hoạt động.

🔴 **Critical: `st.markdown()` bị DOMPurify strip toàn bộ script tag**

Kế hoạch nêu `st.markdown(unsafe_allow_html=True)` như một phương án. Streamlit dùng DOMPurify trước khi render — `<script>` bị xóa sạch. CSS `position:fixed` từ thẻ `<style>` hoạt động, nhưng không có JS thì widget không làm được gì.

> **KẾT LUẬN:**
>
> Loại bỏ phương án `st.markdown()` khỏi kế hoạch. Chỉ dùng `st.components.v1.html()` — đây là cách duy nhất chạy được JS trong Streamlit.

⚠️ **Warning: `position: fixed` bên trong iframe = fixed tương đối với iframe, không phải màn hình**

Khi widget chạy trong iframe của `st.components.v1.html()`, `position: fixed; bottom: 20px; right: 20px` sẽ neo vào góc của iframe container, không phải góc màn hình thực. Kết quả trông "gần đúng" nhưng scroll page sẽ lộ ra sự khác biệt.

> **LỰA CHỌN THỰC TẾ:**
>
> - Cách A: Đặt component ở sidebar Streamlit thay vì floating — đơn giản, luôn hiển thị, không cần hack.
> - Cách B: Dùng thư viện `streamlit-float` (pip install) đã xử lý sẵn vấn đề này.
> - Cách C: Chấp nhận widget cố định ở cuối trang thay vì floating góc màn hình.

⚠️ **Warning: CORS `allow_origins=["*"]` không phù hợp cho production**

Kế hoạch đề xuất mở toàn bộ origins. Trong môi trường dev local thì ổn, nhưng khi deploy hoặc demo sẽ tạo rủi ro bảo mật — bất kỳ trang web nào cũng có thể gọi backend.

> **SỬA NGAY:**
>
> **Python**
>
> ```
> allow_origins=["http://localhost:8501"]  # dev
> allow_origins=[os.getenv("FRONTEND_URL")] # prod
> ```

## Phần 2 — Vi phạm ai-requirement.md

🔴 **Vi phạm §2.1: Auto-execute (tự động gọi `/api/execute` sau khi nhận code) — vi phạm Approval Principle**

Kế hoạch nêu rõ: _"Tự động gọi tiếp /api/execute để chạy code và lấy biểu đồ."_ Yêu cầu §2.1 quy định: _"Code generated by AI will initially be in a pending approval status. Code snippet is ONLY executed if and only if humans approve it."_ Auto-execute bỏ qua toàn bộ cơ chế phê duyệt đã xây dựng trong backend.

> **FLOW ĐÚNG — BẮT BUỘC PHẢI CÓ BƯỚC REVIEW:**
>
> **Plaintext**
>
> ```
> 1. sendMessage() → POST /api/chat → nhận {chat_id, code, explanation}
> 2. Hiển thị code + explanation cho user xem  ← BẮT BUỘC
> 3. User click "Chạy" (approve) hoặc "Bỏ qua"  ← BẮT BUỘC
> 4. Nếu approve → POST /api/execute {code, chat_id}
> 5. Hiển thị kết quả / biểu đồ
> ```

⚠️ **Vi phạm §2.2: History mất khi Streamlit re-render — vi phạm Storage Principle**

Kế hoạch lưu history trong JS array (memory). Streamlit re-render toàn bộ app mỗi khi có tương tác (click button, chọn filter...). Component iframe bị reset → history array về rỗng → mất toàn bộ ngữ cảnh hội thoại. §2.2 yêu cầu _"store all requests... so that they can be retrieved."_

> **HAI LỚP LƯU TRỮ CẦN THIẾT:**
>
> **Plaintext**
>
> ```
> Lớp 1 (session): sessionStorage trong JS → persist qua Streamlit re-renders
>   sessionStorage.setItem('chat_history', JSON.stringify(history))
>
> Lớp 2 (persistent): GET /api/logs → load history từ database khi khởi tạo widget
>   Đảm bảo history không mất dù reload trang
> ```
